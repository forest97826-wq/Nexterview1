一、核心本质（底层逻辑）
Python 装饰器是基于 “万物皆对象” 特性实现的语法糖工具，本质是 “接收一个函数 / 类对象，返回另一个对象（函数 / 类实例）”，核心目标是：在不侵入原代码的前提下，扩展功能或适配外部规范。
二、通用核心作用（所有装饰器都具备）
无侵入扩展功能：不修改原函数 / 类的代码、不改变调用方式，动态给其添加通用功能（如计时、日志、权限校验、重试、缓存等），彻底解耦 “通用逻辑” 和 “核心业务逻辑”，符合 “开闭原则”（对扩展开放、对修改关闭）。
例：给 10 个函数加计时功能，只需写 1 个计时装饰器，无需改每个函数的代码。
复用通用逻辑：通用功能（如日志、校验）只需实现一次，可套在任意函数 / 类上，避免重复写相同代码，符合 “DRY 原则”（Don't Repeat Yourself）。
统一管控规范：对函数 / 类的输入、输出做统一处理（如参数类型校验、返回值格式化），保证代码符合既定规则。
三、框架场景下的专属作用（结合 LangChain 例子）
在 LangChain 等框架中，装饰器额外承担 “适配框架规范” 的核心作用：
类型转换：把普通函数（function类实例）转换成框架指定类的实例，让自定义代码符合框架要求。
例：@tool把普通search函数转成Tool类实例，@wrap_model_call把选模型的函数转成AgentMiddleware类实例。
隐藏复杂适配：框架将参数封装、上下文传递、接口校验等繁琐逻辑藏在装饰器里，开发者只需聚焦核心业务（如写 search 的查询逻辑、选模型的判断逻辑）。
关键记忆点
装饰器的核心价值可浓缩为两句话：
基础场景：给函数 / 类 “加功能不改代码”，通用逻辑只写一次；
框架场景：把自定义代码 “包装成框架认的对象”，不用管适配细节。

第三步：回到装饰器的核心价值总结
装饰器的 “解耦” 和 “符合开闭原则”，本质是一回事：
解耦：把 “通用功能（计时、日志）” 和 “业务逻辑（炒菜）” 拆成两部分，各自独立维护；
开闭原则：要加新功能（比如加权限、加缓存），只加新装饰器就行（扩展开放），不用改业务函数（对修改关闭）。
