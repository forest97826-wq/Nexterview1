# 数据库与中间件面试题

## 一、MySQL

### 1.1 索引相关

**Q：B+ 树为什么适合做索引？**

> B+ 树的特点：所有数据存在叶子节点，叶子节点用链表串联，非叶子节点只存 key 不存数据。
>
> 三个优势：
> 1. **磁盘友好**：非叶子节点不存数据，一个节点能装更多 key，树的高度更低（通常 3-4 层），一次查询只需 3-4 次磁盘 IO
> 2. **范围查询高效**：叶子节点链表串联，范围查询只需找到起点然后顺序遍历，不需要回溯
> 3. **查询性能稳定**：所有查询都要走到叶子节点，时间复杂度稳定 O(log n)

**Q：聚簇索引和非聚簇索引的区别？**

> - **聚簇索引**：叶子节点存的是**完整行数据**。InnoDB 的主键索引就是聚簇索引，数据按主键顺序物理存储。一张表只能有一个聚簇索引
> - **非聚簇索引**（二级索引）：叶子节点存的是**主键值**。查到主键后还要**回表**（用主键去聚簇索引再查一次）拿完整数据
> - **覆盖索引**：如果查询的字段都在二级索引里，不需要回表，叫覆盖索引。Explain 里显示 `Using index`

**Q：联合索引的最左匹配原则？**

> 联合索引 `(a, b, c)` 的存储是先按 a 排序，a 相同按 b 排序，b 相同按 c 排序。
>
> - `WHERE a=1 AND b=2 AND c=3` → 走索引
> - `WHERE a=1 AND b=2` → 走索引
> - `WHERE a=1` → 走索引
> - `WHERE b=2 AND c=3` → **不走索引**（跳过了最左列 a）
> - `WHERE a=1 AND c=3` → 只用到 a，c 用不上（b 被跳过了）

**Q：索引失效的常见场景？**

> 1. 对索引列使用函数或运算：`WHERE YEAR(create_time) = 2025` → 失效
> 2. 隐式类型转换：字段是 varchar，查询 `WHERE phone = 13800138000`（没加引号）→ 失效
> 3. LIKE 左模糊：`WHERE name LIKE '%张'` → 失效；`WHERE name LIKE '张%'` → 走索引
> 4. OR 条件中有非索引列：`WHERE a=1 OR b=2`，如果 b 没索引 → 全表扫描
> 5. 不满足最左匹配原则

### 1.2 Explain 执行计划

**Q：Explain 怎么看，关键字段是什么？**

> ```sql
> EXPLAIN SELECT * FROM users WHERE name = '张三';
> ```
>
> 核心字段：
>
> | 字段 | 含义 | 关注点 |
> |------|------|--------|
> | **type** | 访问类型 | 性能从好到差：`const > eq_ref > ref > range > index > ALL`。看到 ALL（全表扫描）就要优化 |
> | **key** | 实际使用的索引 | NULL 说明没走索引 |
> | **rows** | 预估扫描行数 | 越少越好 |
> | **Extra** | 附加信息 | `Using index`（覆盖索引，好），`Using filesort`（额外排序，差），`Using temporary`（临时表，差） |

**Q：慢查询怎么排查？**

> 1. 开启慢查询日志：`slow_query_log = ON`，设阈值 `long_query_time = 1`（超过1秒记录）
> 2. 找到慢 SQL 后用 `EXPLAIN` 分析执行计划
> 3. 常见优化手段：
>    - 加索引（最常见）
>    - 避免 `SELECT *`，只查需要的字段（减少回表）
>    - 大表分页用延迟关联：`SELECT * FROM t INNER JOIN (SELECT id FROM t LIMIT 100000, 10) AS tmp ON t.id = tmp.id`
>    - 避免在 WHERE 中对索引列做函数运算

### 1.3 事务与锁

**Q：事务的 ACID 是什么？**

> - **A（Atomicity 原子性）**：事务要么全做，要么全不做。通过 undo log 实现回滚
> - **C（Consistency 一致性）**：事务前后数据满足约束。是最终目标，由 AID 共同保证
> - **I（Isolation 隔离性）**：并发事务互不干扰。通过锁 + MVCC 实现
> - **D（Durability 持久性）**：事务提交后数据不丢。通过 redo log 实现

**Q：MySQL 的四种隔离级别？**

> | 隔离级别 | 脏读 | 不可重复读 | 幻读 |
> |---------|------|-----------|------|
> | 读未提交 | 有 | 有 | 有 |
> | 读已提交 | 无 | 有 | 有 |
> | **可重复读（默认）** | 无 | 无 | InnoDB 通过 MVCC+间隙锁基本解决 |
> | 串行化 | 无 | 无 | 无 |
>
> InnoDB 默认可重复读，通过 MVCC（多版本并发控制）实现：每行数据有隐藏的版本号，事务只能看到自己开始前已提交的版本。

**Q：什么是死锁，怎么解决？**

> 两个事务互相等待对方持有的锁。
>
> - InnoDB 有死锁检测机制，检测到后自动回滚代价较小的事务
> - 预防：按固定顺序访问表和行，缩短事务时长，避免大事务

## 二、Redis

### 2.1 基础概念

**Q：Redis 为什么快？**

> 1. **内存操作**：数据在内存中，读写微秒级
> 2. **单线程模型**：避免上下文切换和锁竞争（6.0 后 IO 多线程，但命令执行仍是单线程）
> 3. **IO 多路复用**：epoll 模型，单线程高效处理大量连接
> 4. **高效数据结构**：跳表、压缩列表等针对内存优化的结构

**Q：Redis 常用数据类型？**

> | 类型 | 用途 | 场景 |
> |------|------|------|
> | String | 基本 KV | 缓存、计数器、分布式锁 |
> | Hash | 字段级操作 | 用户信息、对象属性 |
> | List | 有序列表 | 消息队列、最新列表 |
> | Set | 去重集合 | 标签、共同好友 |
> | Sorted Set | 有序去重集合 | 排行榜、热度排名 |

### 2.2 缓存问题

**Q：Redis 怎么给 MySQL 做缓存？**

> **Cache Aside Pattern（旁路缓存）**，在应用层手动维护缓存逻辑，Redis 本身不感知 MySQL。
>
> **读流程**：先查 Redis → 命中直接返回；未命中 → 查 MySQL → 结果写入 Redis（设 TTL）→ 返回
>
> **写流程**：先更新 MySQL → 删除 Redis 对应 key（不是更新，避免并发写导致不一致）
>
> **为什么有效**：数据访问符合二八定律，80% 请求集中在 20% 热数据。MySQL 单机 QPS 几千（磁盘 IO 瓶颈），Redis 单机 QPS 十万级（纯内存）。热数据放 Redis 挡一层，大部分读请求到不了 MySQL。

**Q：缓存穿透、击穿、雪崩的区别和解决方案？**

> **缓存穿透**：查询一个**根本不存在**的数据，缓存和数据库都没有，每次都打到数据库。
> - 解决：布隆过滤器拦截不存在的 key；缓存空值（设短 TTL）
>
> **缓存击穿**：某个**热点 key 过期**的瞬间，大量请求同时打到数据库。
> - 解决：热点 key 永不过期；互斥锁，只让一个请求回源，其他等待
>
> **缓存雪崩**：大量 key **同时过期**，请求全部打到数据库。
> - 解决：TTL 加随机偏移，避免同时过期；多级缓存（本地缓存 + Redis）

**Q：缓存和数据库的一致性怎么保证？**

> 常用策略：**Cache Aside Pattern**（旁路缓存）
> - **读**：先查缓存，命中直接返回；未命中查数据库，写入缓存
> - **写**：先更新数据库，再删除缓存（不是更新缓存）
>
> 为什么是删除缓存而不是更新缓存？因为并发场景下两个写请求可能导致缓存和数据库不一致。删除缓存让下次读去回源，天然一致。
>
> 极端情况（删除缓存失败）：用消息队列重试删除，或设短 TTL 兜底。

### 2.3 分布式锁

**Q：Redis 分布式锁怎么实现？**

> ```
> SET lock_key unique_value NX EX 30
> ```
>
> - `NX`：key 不存在才设置（保证互斥）
> - `EX 30`：30 秒自动过期（防死锁）
> - `unique_value`：每个客户端用唯一值（释放时验证是自己加的锁）
>
> 释放锁时用 Lua 脚本保证原子性：先判断值是否是自己的，是才删除。
>
> ```lua
> if redis.call("get", KEYS[1]) == ARGV[1] then
>     return redis.call("del", KEYS[1])
> else
>     return 0
> end
> ```

**Q：锁过期了但业务没执行完怎么办？**

> **看门狗机制**（Redisson 的实现）：后台线程定期检测锁是否还被持有，如果业务还在执行就自动续期。默认每 10 秒检测一次，续期 30 秒。

### 2.4 持久化

**Q：Redis 的持久化方式？**

> | 方式 | 原理 | 优点 | 缺点 |
> |------|------|------|------|
> | **RDB** | 定时生成内存快照到磁盘 | 恢复速度快，文件紧凑 | 可能丢失最后一次快照后的数据 |
> | **AOF** | 每条写命令追加到日志文件 | 数据安全性高（最多丢 1 秒） | 文件大，恢复慢 |
> | **混合** | RDB 快照 + 增量 AOF | 兼顾恢复速度和数据安全 | Redis 4.0+ 支持 |
>
> 生产环境一般用混合持久化。

### 2.5 Redis 在你项目中的应用

> - **语义缓存**（招生机器人）：embedding 向量 + Redis 向量检索（RediSearch），高频问题命中率 35%，LLM 调用成本降低 30%
> - **MTM 中期记忆**（校园助手）：按话题分段的摘要 + 热度分数，会话开始时加载到内存，结束时写回 Redis
> - **Session 存储**：用户会话状态持久化

## 三、Kafka / RabbitMQ

### 3.1 消息队列基本概念

**Q：什么是消息队列？为什么要用？**

> 消息队列是一个"中间人"，生产者把消息丢进去，消费者从里面取出来处理。三个核心价值：
>
> 1. **异步**：用户下单后，发短信、扣库存、生成物流单不用同步等，丢进队列各自处理
> 2. **削峰**：秒杀 10 万请求，数据库扛不住，先进队列排队，后端按能力消费
> 3. **解耦**：订单服务不需要知道短信服务的存在，只管发消息，谁需要谁订阅

### 3.2 Kafka vs RabbitMQ

**Q：Kafka 和 RabbitMQ 的区别？**

> | 维度 | Kafka | RabbitMQ |
> |------|-------|----------|
> | 核心模型 | 发布-订阅，消息持久化到磁盘，可回溯重放 | 队列模型，消费完消息默认删除 |
> | 吞吐量 | 百万级/秒 | 万级/秒 |
> | 消息顺序 | Partition 内有序 | 队列内有序 |
> | 消费模式 | Pull（消费者主动拉） | Push（broker 推给消费者） |
> | 典型场景 | 日志收集、实时数据管道、事件流 | 任务分发、业务消息传递、通知 |
> | 选型一句话 | **大数据流，需要回溯** | **业务任务分发，需要可靠投递** |

### 3.3 Kafka 核心概念

**Q：Kafka 的 Topic、Partition、Consumer Group 是什么？**

> - **Topic**：消息的类别，比如 `order_events`、`agent_logs`
> - **Partition**：Topic 的分片，每个 Partition 是一个有序的日志文件。消息按 key 哈希分配到不同 Partition，实现并行消费
> - **Consumer Group**：一组消费者共同消费一个 Topic。同一条消息只会被 Group 内的一个消费者处理（负载均衡），不同 Group 各自独立消费（广播）
>
> ```
> Topic: agent_logs
>   ├── Partition 0 → Consumer A（Group 1）
>   ├── Partition 1 → Consumer B（Group 1）
>   └── Partition 2 → Consumer C（Group 1）
>
> 同时 Group 2 的消费者独立消费所有 Partition（用于监控）
> ```

### 3.4 RabbitMQ 核心概念

**Q：RabbitMQ 的 Exchange、Queue、Binding 是什么？**

> - **Exchange**：消息路由器，接收生产者的消息，按规则分发到 Queue
> - **Queue**：消息存储队列，消费者从 Queue 取消息
> - **Binding**：Exchange 和 Queue 之间的路由规则
>
> Exchange 类型：
> - **Direct**：按 routing key 精确匹配，`order.create` → 只发到绑定了这个 key 的队列
> - **Fanout**：广播，发到所有绑定的队列
> - **Topic**：按 routing key 模式匹配，`order.*` 匹配 `order.create`、`order.cancel`

### 3.5 常见面试题

**Q：怎么保证消息不丢？**

> 三个环节都要保证：
> 1. **生产者 → Broker**：Kafka 用 acks=all（所有副本确认）；RabbitMQ 用 confirm 模式
> 2. **Broker 存储**：Kafka 消息写磁盘 + 多副本；RabbitMQ 持久化队列 + 持久化消息
> 3. **Broker → 消费者**：消费者处理完再确认（手动 ack），不要自动 ack

**Q：怎么保证消息不重复消费？**

> 消息队列本身无法保证 exactly-once，需要消费者做**幂等处理**：
> - 数据库唯一约束：重复插入自动失败
> - Redis 记录已消费的消息 ID：处理前先查是否已处理过
> - 业务逻辑天然幂等：比如 `SET balance = 100` 执行多次结果一样

**Q：消息堆积了怎么办？**

> 1. 临时扩容消费者数量
> 2. 如果消费者有 bug 导致消费慢，先修 bug
> 3. Kafka：增加 Partition 数量（消费者数量 ≤ Partition 数量才有意义）
> 4. 设置 TTL，超时的非关键消息直接丢弃

### 3.6 在 Agent 系统中的应用场景

> 大部分 Agent 项目用不到消息队列，只有企业级大规模 Agent 平台才需要：
>
> - **任务调度**（RabbitMQ）：大量用户并发触发 Agent 任务，Worker 按能力排队消费
> - **多 Agent 异步通信**（RabbitMQ）：Agent 之间通过队列解耦，独立扩缩容
> - **日志与监控**（Kafka）：Agent 的每次 LLM 调用、tool call 作为事件推到 Kafka，下游做监控、计费、审计
>
> **面试话术**：
> > "我目前的 Agent 项目日均几百次请求，单进程 FastAPI + asyncio 就够了，没有引入消息队列。但我了解在大规模多 Agent 平台中，RabbitMQ 用于任务调度和 Agent 间通信，Kafka 用于日志流和事件审计。选型上 Kafka 适合高吞吐日志场景，RabbitMQ 适合业务任务分发。"
